# Canal 产品设计实践计划

本文承接 [Canal 产品设计问题清单](canal-product-design-questions.md)，把问题转成可执行的产品设计推进计划。目标不是直接写代码，而是先把角色、链路、任务依赖、风险控制和页面信息架构落到明确产物，避免后续实现反复返工。

## 目标

- 明确 Canal 的产品定位：长期数据链路、一次性迁移任务，或两者兼容。
- 固化创建流程：选择数据源、选择任务类型、选择表、选择列。
- 建立任务依赖模型：结构迁移、全量迁移、增量同步、结构对比、数据校验、数据订正。
- 定义高风险操作边界：预检、数据订正、覆盖写入、审计和回滚提示。
- 输出后续设计和实现可以直接使用的验收标准。

## 修改范围

本节限定后续按本实践计划落地时允许改动的项目范围。所有改动都必须能对应到本文的阶段、产物或验收标准；超出范围时，先更新本计划并确认，再进入实现。

### 文档修改范围

允许修改：

- `docs/canal-product-design-practice-plan.md`：补充阶段、产物、验收标准、排期和风险。
- `docs/canal-product-design-questions.md`：仅在产品决策已经变化时同步修订问题清单。
- `docs/` 下新增决策类文档：产品决策记录、任务组合规则表、预检规则表、状态矩阵、设计验收用例。

不允许修改：

- 与 Canal 创建和管理无关的文档。
- 已归档或已删除的旧 PRD 文档。
- 部署、发布、运维手册，除非本计划明确产出对应要求。

### 前端修改范围

允许修改：

- `frontend/src/App.tsx` 中 Canal 列表、创建向导、详情页、映射、任务、运行、日志相关代码。
- `frontend/src/lib/api.ts` 中 Canal 相关 API 调用。
- `frontend/src/types/api.ts` 中 Canal、Task、Run、Log、Precheck、Diff 相关类型。
- `frontend/src/components/ui.tsx` 中复用型表单、按钮、弹窗、提示组件，前提是 Canal 页面现有组件无法满足。
- `frontend/src/index.css` 中与 Canal 页面布局或控件状态直接相关的少量样式。

不允许修改：

- 全站导航结构，除非只涉及 `/canals` 入口。
- 无关页面、无关业务组件和无关样式主题。
- 大规模设计系统重构。
- 路由体系重命名，除非是维护 `/canals` 兼容性。

### 后端修改范围

允许修改：

- `backend/internal/app/types.go` 中 Canal、Task、Run、Log、Precheck、DataValidationDiff 相关模型。
- `backend/internal/app/channel_server.go` 中 Canal 相关接口处理。
- `backend/internal/app/channel_store.go` 和 `backend/internal/app/channel_store_test.go` 中 Canal 相关存储、校验和测试。
- `backend/internal/app/store_persistence_models.go` 和 `backend/internal/app/store_persistence.go` 中 Canal 相关持久化结构。
- 数据源元数据读取接口，前提是为了支撑 DB、schema、table、column 选择。

不允许修改：

- DataX 类全量迁移引擎实现。
- Canal 类增量同步引擎实现。
- 数据订正执行引擎。
- 调度平台、审批系统、告警系统。
- 用户权限体系，除非仅补齐 Canal 操作所需的最小判断。
- 与 Canal 无关的后端资源、接口和持久化模型。

### 阶段到修改范围

- 阶段 0：只改 `docs/`。
- 阶段 1：只读代码和页面，输出盘点文档；默认不改功能代码。
- 阶段 2：允许改 Canal 创建页、Canal 相关 API 类型和必要的元数据接口。
- 阶段 3：允许改预检接口、预检展示、任务启动阻断和对应测试。
- 阶段 4：允许改 Canal 列表和详情页信息架构，不改无关页面。
- 阶段 5：允许补充 Diff 模型、校验结果展示和订正确认规则，不实现真实订正引擎。
- 阶段 6：允许统一 Canal、Task、Run、Log 状态展示和日志筛选。
- 阶段 7：允许新增验收用例和必要测试，不扩展新业务能力。

### 变更颗粒度

- 每次实现只选择一个阶段或一个明确子任务。
- 每次提交只包含该子任务相关文件。
- 每次变更必须同步更新对应验收标准。
- 涉及高风险操作时，先补规则和文档，再改交互或接口。
- 如果需要跨出上述范围，先把原因、影响文件和验收方式写回本计划。

## 当前基座

当前项目已经具备以下产品骨架：

- 创建入口：`/canals/create`。
- 创建步骤：选择数据源、选择任务类型、选择表、选择列。
- 详情页标签：概览、映射、任务、运行、日志。
- 核心对象：Canal、Task、Run、Log、Precheck、Mapping、DataValidationDiff。
- 任务类型：结构迁移、全量迁移、增量同步、结构对比、数据校验、数据订正。
- 状态类型：Canal 状态、Task 状态、Run 状态已分层。

实践计划应优先复用这些已有结构，不重新设计一套产品模型。

## 设计原则

- 先决策，后扩展：先确认高影响问题，再补页面能力。
- 先闭环，后批量：先跑通单 Canal、单链路，再做批量表和批量规则。
- 自动优先，人工兜底：DB、表、列、主键尽量自动读取，用户只确认和修正。
- 风险显性：数据订正、覆盖写入、跳过预检必须被明确表达。
- 状态统一：列表、详情、任务、运行、日志使用同一套状态语言。
- 文案克制：页面只保留必要标签、状态和错误，减少说明型文字。

## 阶段 0：决策校准

时间建议：0.5 到 1 天。

目标：把会影响信息架构和任务模型的问题先定下来。

需要确认：

- 核心用户是 DBA、数据开发、运维，还是业务负责人。
- Canal 默认是长期链路，还是一次性任务集合。
- 同步型和检查型是否共用同一个创建流程。
- 六类任务的组合规则和依赖顺序。
- 数据订正是否必须绑定数据校验结果。
- 创建后默认进入“待启动”还是自动启动。

产物：

- `Canal 产品决策记录`。
- `任务组合规则表`。
- `高风险操作清单`。

验收标准：

- 每个决策都有结论、原因、影响页面和待补能力。
- 结构迁移、全量迁移、增量同步、结构对比、数据校验、数据订正的依赖关系无歧义。
- 数据订正的入口、权限、确认和审计边界有明确结论。

## 阶段 1：现有流程盘点

时间建议：0.5 天。

目标：把当前页面与目标体验之间的差距列清楚。

盘点范围：

- Canal 创建页四步表单。
- Canal 列表页。
- Canal 详情页概览。
- 映射、任务、运行、日志标签页。
- 后端对象和接口能力。

重点检查：

- 每一步是否有明确完成条件。
- 用户失败后是否知道下一步动作。
- 数据源测试、DB 拉取、表拉取、列配置之间是否顺滑。
- 详情页是否能回答“现在能不能启动”和“失败在哪里”。
- 任务状态和 Canal 状态是否可能互相矛盾。

产物：

- `现有流程问题清单`。
- `页面到对象映射表`。
- `接口能力缺口表`。

验收标准：

- 每个问题都能落到页面、接口或数据模型。
- 每个缺口都有优先级：P0、P1、P2。
- 不把引擎能力缺口误标成 UI 问题。

## 阶段 2：创建流程定稿

时间建议：1 到 2 天。

目标：把 `/canals/create` 定成稳定的 V1 创建体验。

### 第一步：选择数据源

设计要求：

- 页面上方横向展示四步进度。
- 运行节点放在名称下方。
- 数据源类型使用按钮组或卡片选择，不使用下拉框。
- 源端和目标端分别选择数据源。
- 点击测试连接后，成功在按钮旁展示“测试连接成功”。
- 测试失败弹窗展示失败原因。
- 测试成功后自动加载 DB。

待定规则：

- MySQL 隐藏 schema。
- 非 MySQL 是否显示 schema，由数据源类型能力决定。
- 测试连接是否只验证连接，还是同时验证元数据权限。

验收标准：

- 未选运行节点时不能测试连接。
- 源端和目标端不能选择同一个数据源。
- 测试成功前不能进入表选择。
- 连接失败原因可被复制或完整查看。

### 第二步：选择任务类型

设计要求：

- 先选择 Canal 类型：同步或检查。
- 同步型包含：结构迁移、全量迁移、增量同步。
- 检查型包含：结构对比、数据校验、数据订正。
- 结构迁移作为同步型基础能力展示。
- 数据订正依赖数据校验，未选数据校验时不可选。

待定规则：

- 增量同步是否必须依赖全量迁移。
- 结构迁移是否允许关闭。
- 结构对比是否允许作为独立 Canal 创建。
- 资源规格是否继续由用户选择，还是由系统推荐。

验收标准：

- 用户能看懂任务执行顺序。
- 不允许提交无效组合。
- 高风险组合在提交前有预检或确认。

### 第三步：选择表

设计要求：

- DB、schema、table 都由用户通过下拉框选择。
- MySQL 类型不展示 schema。
- 源端和目标端分别选择 DB。
- 源表、目标表通过下拉框选择。
- 支持主键填写或自动识别。
- 支持是否创建目标表。

待定规则：

- 目标表是否默认同名。
- 目标表不存在时，是允许建表、提示缺失，还是阻断。
- 主键是否必须来自源端自动识别。

验收标准：

- DB 改变后，表选择和列选择自动重置。
- 数据校验和数据订正场景必须有主键。
- 表级映射可以保存为稳定版本。

### 第四步：选择列

设计要求：

- 默认生成同名列映射。
- 支持源列、源类型、目标列、目标类型、主键、可空。
- 支持单列新增和删除。
- 标出类型不一致、主键缺失、目标列缺失。

待定规则：

- 是否支持大写转换、前后缀、字段改名规则。
- 类型不一致时是阻断还是警告。
- 默认值是否纳入结构迁移。

验收标准：

- 至少一列有效映射才允许创建。
- 主键字段能和表级主键保持同步。
- 列映射结果能用于结构迁移、全量迁移、数据校验和数据订正。

产物：

- `创建流程规格说明`。
- `四步表单状态机`。
- `字段级验收清单`。

## 阶段 3：预检与风险控制

时间建议：1 天。

目标：把“创建成功”和“可以安全执行”拆开。

预检范围：

- 源端连接。
- 目标端连接。
- DB、schema、表是否存在。
- 目标表是否缺失。
- 列是否缺失。
- 类型是否兼容。
- 主键是否完整。
- 任务依赖是否完整。
- 数据订正是否有校验结果。

设计要求：

- 创建后默认进入详情页。
- 详情页提供预检入口。
- 预检结果按风险级别展示。
- 阻断项不能启动任务。
- 警告项可以继续，但需要留痕。

高风险确认：

- 数据订正。
- replace 写入目标端。
- 跳过警告项继续执行。
- 重新运行已成功任务。
- 删除或归档 Canal。

产物：

- `预检规则表`。
- `风险确认文案`。
- `审计字段清单`。

验收标准：

- 每条预检结果都有 key、label、success、message。
- 用户能从预检结果直接定位到映射、任务或数据源问题。
- 数据订正没有校验结果时不能启动。

## 阶段 4：详情页信息架构

时间建议：1 到 2 天。

目标：让 Canal 详情页成为运行和排障的主入口。

首屏需要回答：

- 当前 Canal 是否健康。
- 是否可以启动。
- 最近一次运行结果是什么。
- 是否有阻断问题。
- 下一步建议动作是什么。

建议信息架构：

- 概览：链路、健康、最近运行、预检摘要、下一步动作。
- 映射：表映射、列映射、版本、保存。
- 任务：任务列表、依赖、启动、停止、重跑。
- 运行：Run 列表、读写行数、失败行数、差异行数。
- 日志：按任务、Run、级别过滤。
- 差异：数据校验 diff 和订正批次。

当前已有概览、映射、任务、运行、日志；差异可以作为数据校验和数据订正闭环的后续扩展。

产物：

- `详情页信息架构图`。
- `状态展示规则`。
- `排障路径说明`。

验收标准：

- 列表状态、详情状态、任务状态、运行状态口径一致。
- 失败场景首屏可见错误摘要。
- 日志能追溯到具体任务和 Run。

## 阶段 5：数据校验与数据订正闭环

时间建议：1 到 2 天。

目标：让数据校验结果可以安全地进入数据订正。

数据校验设计：

- 只支持有主键的表。
- 按表映射和列映射逐行对比。
- 差异结果按主键落盘。
- 结果包含源值、目标值、差异列、校验 Run。

数据订正设计：

- 必须选择一次数据校验结果。
- 根据主键重新查询源端数据。
- 按映射转换后 replace 到目标端。
- 订正前展示影响行数和样例 diff。
- 订正后生成订正 Run 和审计日志。

风险边界：

- 如果目标端数据在校验后被更新，需要提示可能覆盖。
- 如果订正部分失败，需要保留失败主键。
- 如果没有回滚能力，界面必须明确不可逆。

产物：

- `Diff 结果模型`。
- `订正流程说明`。
- `订正确认规则`。

验收标准：

- 无主键表不能创建数据校验或数据订正任务。
- 数据订正不能脱离校验结果启动。
- 每次订正都能追踪到来源校验 Run。

## 阶段 6：状态语言与错误体系

时间建议：0.5 到 1 天。

目标：统一 Canal、Task、Run、Log 的状态和错误表达。

需要定义：

- Canal 状态如何由 Task 和 Run 汇总。
- Task 状态和 Run 状态如何区分。
- warning、failed、stopped、archived 的优先级。
- 错误摘要和原始日志分别放在哪里。
- 日志过滤维度。

建议口径：

- Canal 表示整体链路健康。
- Task 表示任务配置和执行开关。
- Run 表示一次执行结果。
- Log 表示排障细节。
- Diff 表示数据不一致事实。

产物：

- `状态矩阵`。
- `错误展示规范`。
- `日志筛选规则`。

验收标准：

- 同一个错误不会在不同页面显示成不同状态。
- 列表只显示摘要，详情提供原始日志。
- 运行节点、任务、Run 都可追踪。

## 阶段 7：设计验收与迭代发布

时间建议：1 天。

目标：用真实场景验收计划是否可落地。

验收场景：

- 结构迁移：A 表映射成 B 表，字段 a/b/c 映射成 A/B/C。
- 全量迁移：同一套映射迁移数据到目标端。
- 增量同步：全量后持续运行。
- 结构对比：目标端缺列时输出日志。
- 数据校验：有主键表逐行对比并落盘 diff。
- 数据订正：基于 diff 按主键 replace 到目标端。

设计检查：

- 四步创建在桌面和窄屏都可用。
- 所有按钮文案短且明确。
- 高风险操作有确认和审计。
- 错误信息能指导下一步。
- 空状态不堆解释文字。

产物：

- `设计验收用例`。
- `发布检查清单`。
- `后续迭代 Backlog`。

验收标准：

- 六类任务都有完整用户路径。
- P0 风险场景都有阻断或确认。
- 文档、页面、接口对象命名一致。

## 优先级拆分

P0：

- Canal 定位和用户角色。
- 任务依赖模型。
- 数据订正风险边界。
- 四步创建流程完成条件。
- 预检规则。
- 状态矩阵。

P1：

- DB、schema、table 下拉联动。
- 同名表和同名列默认映射。
- 主键自动识别。
- 详情页首屏下一步动作。
- 日志过滤。
- Diff 结果入口。

P2：

- 批量表映射。
- 列名大小写批量转换。
- 审批流。
- 回滚策略。
- 多源多目标映射。
- 可视化任务依赖图。

## 建议交付顺序

1. 完成决策记录和任务依赖模型。
2. 定稿创建流程规格。
3. 补齐预检规则和风险确认。
4. 定稿详情页信息架构。
5. 定义数据校验和数据订正闭环。
6. 输出状态矩阵和验收用例。
7. 再进入页面实现或交互细化。

## 不在本轮解决

- DataX 或 Canal 引擎级实现。
- 生产级权限审批系统。
- 跨项目任务调度平台。
- 全量性能优化策略。
- 数据订正回滚引擎。

这些内容可以进入后续技术方案或架构文档，不应阻塞当前产品设计定稿。
