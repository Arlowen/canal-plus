## 2. 背景与问题

Canal 能基于 MySQL binlog 做增量数据同步，但实际使用门槛偏高。用户往往需要理解 instance、adapter、destination、groupId、filter、mapping 等概念，并维护多份配置文件。对只想完成“把一个 MySQL 表同步到另一个目标”的用户来说，当前流程过重，问题主要体现在：

- 配置分散：数据源、目标端、表过滤、字段映射、运行参数分布在多份文件中。
- 上手困难：用户必须先理解 Canal 内部模型，才能完成一个简单任务。
- 变更成本高：新增表、修改字段映射、暂停恢复任务通常需要改配置、重启或重新部署。
- 可观测性弱：同步延迟、错误记录、binlog 位点、失败重试状态不够直观。
- 运维依赖强：普通开发或业务用户难以自助完成同步链路创建、排错和日常维护。

Canal Plus 的目标是把 MySQL 同步从“配置文件驱动”升级为“可视化任务驱动”，让用户通过前端向导完成数据源接入、目标配置、表选择、字段映射、任务发布、监控告警和问题处理。

## 3. 产品定位

Canal Plus 是一个面向开发、数据工程和中小团队的 MySQL CDC 同步管理平台，提供前后端一体化控制台，用更简单的交互完成 MySQL binlog 同步任务的创建、运行和运维。

产品核心不是替代所有数据集成平台，而是先做好高频场景：

- MySQL 到 MySQL 的库表同步。
- MySQL 到 Elasticsearch 的搜索索引同步。
- MySQL 到 Kafka 的事件分发。
- 后续扩展 HTTP Webhook、Doris、ClickHouse、PostgreSQL 等目标端。

MVP 优先聚焦 MySQL 到 MySQL，以保证核心链路稳定、操作简单、可验证。

## 4. 目标用户

### 4.1 开发工程师

负责业务系统数据同步，希望快速把业务库的一批表同步到查询库、报表库或测试环境。诉求是少配置、可回滚、错误能看懂。

### 4.2 数据工程师

负责维护数据链路，希望统一管理多个同步任务，关注任务健康度、同步延迟、失败重试和字段变更影响。

### 4.3 运维或平台工程师

负责部署和资源管理，希望平台具备统一配置、权限控制、告警通知、运行日志和可审计操作。

## 5. 产品目标

### 5.1 业务目标

- 让用户在 5 分钟内完成一个 MySQL 到 MySQL 的同步任务创建。
- 让用户无需手写 Canal 配置文件即可完成任务发布、暂停、恢复和删除。
- 将常见同步问题通过页面展示为可理解、可定位、可处理的状态。
- 形成可扩展的同步任务模型，为后续目标端扩展打基础。

### 5.2 非目标

MVP 阶段暂不追求：

- 复杂实时计算能力。
- 全量替代 Flink CDC、DataX、Debezium 或企业级数据集成平台。
- 跨地域多活同步和强一致分布式事务。
- 自动处理所有复杂 DDL 变更。
- 面向海量租户的 SaaS 计费和租户隔离。

## 6. 核心价值主张

用一个控制台完成 MySQL 同步任务的全生命周期管理：

- 向导式创建：连接数据源、选择库表、配置目标、确认映射、发布任务。
- 配置集中化：所有任务配置存储在数据库中，页面可查看、可修改、可审计。
- 状态可视化：任务运行状态、同步延迟、binlog 位点、错误重试、吞吐量清晰展示。
- 运维低门槛：支持暂停、恢复、重跑、跳过错误、查看错误详情。
- 可扩展架构：目标端和同步策略以插件或适配器方式扩展。

## 8. 典型用户流程

### 8.1 创建同步任务

1. 用户进入“同步任务”页面，点击“新建任务”。
2. 选择源 MySQL 数据源，或新建数据源。
3. 平台测试连接，并读取库、表、字段和主键信息。
4. 用户选择需要同步的库表。
5. 选择目标 MySQL，配置目标库和目标表策略。
6. 平台生成默认字段映射。
7. 用户确认主键、字段映射、写入模式和冲突策略。
8. 用户选择是否执行全量初始化。
9. 用户确认任务配置并发布。
10. 平台创建任务，启动全量同步，完成后自动进入增量同步。

### 8.2 处理同步错误

1. 用户在任务列表看到任务状态为“异常”。
2. 进入任务详情，查看失败 SQL、错误原因、表名、主键、binlog 位点。
3. 用户选择“重试”“跳过当前事件”或“暂停任务”。
4. 平台记录用户操作，并更新任务状态。
5. 若错误恢复，任务回到运行中状态。

### 8.3 修改任务配置

1. 用户暂停任务。
2. 修改字段映射、过滤条件或目标写入策略。
3. 平台执行配置校验。
4. 用户保存并恢复任务。
5. 平台从最近成功位点继续同步。

## 9. 功能需求

### 9.1 登录与权限

#### 功能说明

提供基础账号登录和角色权限，避免所有用户都能操作关键同步任务。

#### 需求点

- 支持账号密码登录。
- 支持管理员和普通用户两类角色。
- 管理员可管理数据源、任务、系统配置和用户。
- 普通用户可查看任务，按授权创建和操作任务。
- 所有关键操作需要记录操作日志。

#### 验收标准

- 未登录用户不能访问控制台。
- 普通用户不能删除他人无权限的任务。
- 删除、暂停、跳过错误等操作能在操作日志中追溯到操作者。

### 9.2 数据源管理

#### 功能说明

集中管理源 MySQL 和目标 MySQL 连接信息。

#### 需求点

- 支持新增、编辑、删除、查看数据源。
- 支持连接测试。
- 支持读取数据库、表、字段、主键、索引信息。
- 密码必须加密存储。
- 支持标记数据源用途：源端、目标端、源端和目标端。
- 展示数据源连接状态和最近测试时间。

#### 验收标准

- 用户填写 host、port、username、password 后可测试连接。
- 连接成功后能读取库表结构。
- 数据源被任务引用时不能直接删除。

### 9.3 同步任务创建向导

#### 功能说明

通过分步表单替代手写配置文件。

#### 需求点

- 第一步：填写任务名称、描述、负责人。
- 第二步：选择源数据源和库表。
- 第三步：选择目标数据源和目标库表策略。
- 第四步：配置字段映射。
- 第五步：配置同步策略。
- 第六步：预览配置并发布。

#### 同步策略

- 初始化策略：仅增量、先全量后增量。
- 写入模式：insert/update/delete 同步开关。
- 冲突策略：覆盖、忽略、失败停止。
- 删除策略：物理删除、软删除字段更新、忽略删除。
- 批量写入大小。
- 失败重试次数和重试间隔。

#### 验收标准

- 用户不需要编辑任何配置文件即可创建任务。
- 必填项缺失时页面有明确提示。
- 配置发布前能看到完整任务预览。

### 9.4 表与字段映射

#### 功能说明

支持源表到目标表、源字段到目标字段的映射配置。

#### 需求点

- 默认按同名表、同名字段自动映射。
- 支持修改目标表名。
- 支持字段改名。
- 支持忽略指定字段。
- 支持指定主键字段。
- 支持简单常量字段，例如固定租户 ID。
- 校验目标表是否存在。
- 目标表不存在时支持生成建表 SQL。

#### 验收标准

- 字段类型不兼容时页面给出提醒。
- 主键缺失时不能发布需要 update/delete 的同步任务。
- 用户可复制平台生成的建表 SQL。

### 9.5 全量初始化同步

#### 功能说明

在增量同步前，将源表已有数据批量复制到目标端。

#### 需求点

- 支持按主键或分页批量读取。
- 支持配置批次大小。
- 支持展示全量同步进度。
- 支持失败重试。
- 支持中断后继续。
- 支持全量完成后自动切换到增量同步。

#### 验收标准

- 任务详情能展示总行数、已同步行数、进度百分比。
- 全量同步失败后可重试。
- 全量同步完成后增量同步不丢失期间产生的变更。

### 9.6 增量同步

#### 功能说明

基于 MySQL binlog 捕获 insert、update、delete 事件并写入目标端。

#### 需求点

- 支持从指定 binlog 文件和 position 启动。
- 支持保存最新成功消费位点。
- 支持任务重启后从保存位点恢复。
- 支持按库表过滤事件。
- 支持 DML 事件转换为目标端写入。
- 支持异常事件进入错误队列。

#### 验收标准

- 任务重启后不会从头同步。
- 页面能展示当前 binlog 文件、position、同步延迟。
- 同步异常不会导致整个平台不可用。

### 9.7 任务管理

#### 功能说明

管理同步任务生命周期。

#### 需求点

- 任务状态：草稿、待启动、全量同步中、增量同步中、暂停、异常、已停止。
- 支持启动、暂停、恢复、停止、删除。
- 支持复制任务。
- 支持按名称、状态、负责人、数据源筛选。
- 支持查看任务配置快照。
- 支持查看任务变更历史。

#### 验收标准

- 暂停任务后不再继续消费新事件。
- 恢复任务后从最近成功位点继续。
- 删除任务前需要二次确认。

### 9.8 监控看板

#### 功能说明

提供任务运行状态和核心指标展示。

#### 指标

- 任务总数。
- 运行中任务数。
- 异常任务数。
- 平均同步延迟。
- 每秒处理事件数。
- 最近 24 小时失败次数。
- 全量同步进度。
- binlog 消费位点。

#### 验收标准

- 用户进入首页即可看到异常任务和延迟最高的任务。
- 任务详情页能看到最近一段时间的吞吐和错误趋势。

### 9.9 错误处理

#### 功能说明

将同步错误结构化展示，并提供可控处理入口。

#### 需求点

- 记录错误时间、任务、源表、目标表、事件类型、主键、错误原因、原始事件摘要。
- 支持按错误状态筛选：待处理、已重试、已跳过、已解决。
- 支持单条重试。
- 支持批量重试。
- 支持跳过当前错误事件。
- 跳过操作需要记录操作者和原因。

#### 验收标准

- 用户能从错误详情定位到具体任务和表。
- 跳过事件必须填写原因。
- 重试成功后错误状态变为已解决。

### 9.10 告警通知

#### 功能说明

当任务异常或延迟超过阈值时通知负责人。

#### 需求点

- 支持任务失败告警。
- 支持同步延迟超过阈值告警。
- 支持错误次数超过阈值告警。
- 支持通知渠道预留：企业微信、钉钉、Slack、Webhook。
- MVP 可先实现 Webhook。

#### 验收标准

- 任务异常后能触发一次告警。
- 告警恢复后能记录恢复事件。
- 同一异常需要有基础防抖，避免无限刷屏。

## 11. 后端需求

### 11.1 核心模块

- API 服务：提供前端接口、鉴权和参数校验。
- 元数据服务：管理数据源、任务、映射、配置版本。
- 调度服务：负责任务启动、暂停、恢复、停止。
- 全量同步引擎：负责源表批量读取和目标写入。
- 增量同步引擎：负责 binlog 事件消费、转换和投递。
- 写入适配器：负责不同目标端写入。
- 错误队列：负责失败事件记录、重试和跳过。
- 监控服务：采集任务状态、延迟、吞吐和错误指标。
- 告警服务：根据规则发送通知。

### 11.2 建议架构

MVP 可采用单体后端加内部模块化设计，避免过早拆分微服务。

- 前端：React 或 Vue。
- 后端：Java Spring Boot 或 Go。
- 元数据库：MySQL。
- 任务运行：后端进程内 worker。
- 日志：结构化日志。
- 指标：Prometheus 兼容指标预留。

后续当任务量增长时，可以将 worker 独立为多实例执行器，由控制台统一调度。

### 11.3 数据存储建议

需要至少维护以下核心表：

- user：用户。
- datasource：数据源。
- sync_task：同步任务主表。
- sync_task_config_version：任务配置版本。
- table_mapping：表映射。
- field_mapping：字段映射。
- task_runtime_state：任务运行状态。
- task_checkpoint：binlog 位点。
- task_error_event：错误事件。
- task_operation_log：操作日志。
- alert_rule：告警规则。
- alert_event：告警事件。

## 13. 非功能需求

### 13.1 易用性

- 用户不需要手动编辑配置文件。
- 页面文案应使用“数据源、同步任务、字段映射、错误处理”等业务概念。
- 表单错误需要指出具体字段和修复建议。

### 13.2 可靠性

- 位点持久化必须与目标写入结果保持一致性边界。
- 任务重启后应尽量避免重复或丢失。
- 错误事件应可追踪、可重试、可跳过。
- 后端服务重启不能导致任务配置丢失。

### 13.3 性能

- MVP 单任务目标吞吐：每秒 500 到 2000 条 DML 事件，具体取决于目标库性能。
- 全量同步支持批量读取和批量写入。
- 页面列表查询需要分页。
- 大字段和大事务需要有限流或保护策略。

### 13.4 安全

- 数据源密码加密存储。
- API 需要鉴权。
- 敏感字段在日志中脱敏。
- 用户操作需要审计。
- 禁止前端直接返回明文密码。

### 13.5 可观测性

- 每个任务应有唯一 taskId。
- 日志包含 taskId、table、eventType、checkpoint。
- 关键指标可通过 API 获取，后续可接入 Prometheus。

## 14. 关键产品决策

### 14.1 是否复用 Canal

建议 MVP 技术选型保持开放，但产品上不暴露 Canal 配置概念。

可选方案：

- 封装 Canal：复用成熟 binlog 消费能力，平台生成配置并管理生命周期。
- 自研轻量 binlog consumer：减少外部配置复杂度，但需要自己承担协议和边界问题。
- 基于 Debezium Embedded：生态成熟，但运行模型和依赖更重。

PRD 层面的要求是：无论底层使用哪种方案，用户都只面对 Canal Plus 的数据源、任务、映射和监控模型。

### 14.2 MVP 目标端

建议第一版只做 MySQL 到 MySQL。

原因：

- 最容易验证端到端链路。
- 字段类型和写入语义相对清晰。
- 业务价值明确。
- 后续扩展 Elasticsearch、Kafka 时可以复用任务模型和监控模型。

## 15. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| MySQL DDL 变更复杂 | 字段映射失效、同步失败 | MVP 先提示并暂停任务，后续支持自动兼容 |
| 位点和目标写入一致性难 | 可能重复或丢失数据 | 明确至少一次投递语义，依赖主键幂等写入 |
| 大事务或大表初始化慢 | 任务阻塞、资源占用高 | 批量处理、限流、进度展示、可暂停 |
| 目标库写入失败 | 延迟升高、任务异常 | 错误队列、重试、告警 |
| 用户误操作跳过事件 | 数据不一致 | 二次确认、填写原因、操作审计 |

## 17. MVP 验收标准

MVP 可上线的最低标准：

- 用户可通过页面创建 MySQL 到 MySQL 同步任务。
- 任务可完成一次全量初始化，并自动进入增量同步。
- insert、update、delete 能正确同步到目标表。
- 任务暂停、恢复、停止行为符合预期。
- 后端重启后任务可从 checkpoint 继续。
- 任务异常时页面能展示错误原因，并支持重试或跳过。
- 用户无需手动维护任何 Canal 配置文件。
- 所有数据源密码不以明文形式返回前端。
